# 把 Dialog Jumper 改成「多侧栏」实现上难不难？

现有菜单栏工具在系统 Open/Save 文件对话框旁挂 **一块** 侧栏做文件夹 Jump（Accessibility；不点 Open/Save）。列表数据（最近/收藏/外部目录源）已是全局一份。

若改成：**每个仍打开的标准文件对话框旁边各挂一块侧栏**——生命周期跟该对话框走；列表仍全局共享；只有 Path 输入按侧栏分开；在某块侧栏上 Jump 必须只作用到它对应的那个对话框——

**实现难度如何？主要坑会在哪？有没有低估的点？**

---

## 现状

- 用辅助功能找到系统「打开与存储面板」类进程，确认有可见大窗，再认成标准文件对话框；非系统自绘选择器通常故意不碰。
- 多个对话框同时存在时，会收敛成只服务一个（偏前台宿主），因此全局只有一套侧栏状态机，Jump 目标是隐式的「当前这一个」。
- Jump = 对目标面板进程发 Go to Folder 键序列 + AX 填路径 + 回车；能力上可以打到指定进程，今天只是没把「目标」做成多实例参数。
- 侧栏是 floating panel；几何跟面板 frame；列表仓库与侧栏个数无关。

## 改造在工程上等于什么

```
检测：0..1 个目标  →  集合 = 全部合格 panel
chrome：1 个控制器 →  Map<panel 身份, 侧栏实例> 每 tick 对账（增/删/更新 frame）
Path：全局一个    →  每实例一个字符串
Jump：隐式当前目标 →  Jump(path, panel 身份) 必须显式，事件来自哪块侧栏就打谁
列表数据：         →  继续单例共享
```

显示策略（panel 在就显示 / 仅宿主前台显示）会影响实现量，但不改变「Map + 显式 Jump」这条主线。

## 自认可能难的点（可拍砖）

1. **串台 Jump** — 并排两个 Open，左边点 Jump 进了右边。  
2. **生命周期对账** — Cancel 后进程还在窗没了、检测抖动导致闪/漏。  
3. **panel 身份** — 仅 PID 是否够，还是要到窗口级。  
4. **手测** — 双开系统面板几乎只能人手测，回归成本高于「再加一个列表数据源」。

不觉得难的：多开几个 floating panel 本身；列表继续全局；不依赖未公开 API。

粗感觉：有现成单侧栏 + Jump 的前提下，属中等重构；验证成本可能高于代码量。
