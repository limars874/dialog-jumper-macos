# 多侧栏（每 File Dialog 一块 chrome）技术实现难度说明

Date: 2026-07-12  
给谁看：有 macOS / 桌面 GUI 经验的工程师，帮忙判断 **难不难做、难在哪**  
不是产品该不该做的问卷。

---

## 背景（实现语境，尽量短）

现有程序是一个 **菜单栏 agent 风格** 的 macOS app：

- 用 **Accessibility** 发现系统 **Open/Save 面板服务** 上的标准文件对话框  
- 用 **一个** floating `NSPanel` 当侧栏，吸附在「当前选中的那一个」dialog 旁  
- Jump：对 **该 panel 进程** 做快捷键 + AX 填路径 + 回车（不点 Open/Save）  
- 列表数据（最近/收藏/外部目录源）已是 **进程内全局单例**；侧栏主要是 chrome + 一个 Path 字符串  

目标改造（实现合同假设）：

- 同时存在 N 个合格 File Dialog → **N 个侧栏**，各贴各的  
- Dialog 生命周期 ↔ 对应侧栏 create/destroy（或 hide/show）  
- 列表数据 **继续全局共享**  
- **仅 Path 输入状态** 按侧栏实例分开  
- 在侧栏 A 上 Jump → **只能** 作用到 dialog A（按 panel 进程身份绑定）

---

## 结论先讲

| 判断 | 说明 |
| --- | --- |
| **能不能做** | 能。不依赖未公开 API 幻想；都是 AppKit 多 panel + 现有 AX Jump 的参数化 |
| ** ent 不颠覆** | 中等。不是重写产品，是 **chrome 从 1→N + Jump 显式绑定目标** |
| **真正难点** | 不是「多 new 几个 NSPanel」，而是 **目标进程绑定、生命周期对账、并发 UI 状态** |
| **风险等级** | 做错会 **静默 Jump 到错误 dialog**（正确性 bug），比纯 UI 难测 |
| **粗工作量** | 有现成单侧栏与 Jump 的前提下，熟练 macOS 开发大约 **数天级** 可做出可手测版本；做稳（对账/双开回归）再加时间 |

若团队已熟悉本项目的检测与 Jump：**偏「扎实重构」而不是「探索未知平台能力」。**

---

## 现状实现结构（抽象）

```
检测循环 ──► 0 或 1 个目标 panel（进程 ID / 窗口身份）
                │
                ▼
         单个侧栏控制器
          · 一个 Path 字符串
          · 几何：跟那一个 panel 的 frame
          · Jump：隐式使用「当前这一个」目标
                │
全局列表仓库 ◄──┘  （最近/收藏/… 与侧栏数量无关）
```

多 dialog 时：在检测层 **收敛成一个** 目标，所以永远只有一套 chrome 状态机。

---

## 改造后结构（抽象）

```
检测循环 ──► 集合 S = {panel₁, panel₂, …}
                │
                ▼
     对账：S 与 Map<panelKey, 侧栏实例>
          · 新出现 → create
          · 消失   → destroy
          · 仍在   → 更新 frame / 可见性
                │
     每个侧栏实例
          · 自有 Path 字符串
          · 固定 panelKey（进程+窗口身份）
          · Jump(path, panelKey)  显式
                │
全局列表仓库 ──► 任意侧栏只读同一份；写 Path/Jump 走「事件来源实例」
```

---

## 技术难点拆解（按难度）

### 1. Jump 目标绑定（高 · 正确性核心）

**现在：** Jump 管道默认「全局当前 panel」。  
**多侧栏：** 必须变成 `Jump(path, targetPanel)`，且整条 AX/快捷键链只操作该进程的 UI 树。

难点：

- 侧栏回调必须 **捕获不可变的 target 身份**，不能再读「全局 current」  
- 检测刷新时若重建对象，identity 要稳定（PID 复用、窗口替换）  
- 回归：两个 Open 并排，左 Jump / 右 Jump 交叉点，看是否串台  

**失败模式：** 看起来 UI 对，目录跳错对话框——难用截图一眼发现。

### 2. 检测从「选一个」改为「枚举集合」（中）

**现在：** 扫描后启发式选 frontmost 相关的一个。  
**多侧栏：** 返回 **全部** 合格 panel，去重。

难点：

- 合格条件与现在类似（面板服务 + 可见大窗 + 指纹），主要是 **不要 early-return 只留一个**  
- 同一服务进程多窗时，要定义 key：仅 PID 是否够，还是 PID+window 身份  
- 集合抖动（短暂认不出）会导致侧栏闪断 → 需要短暂宽限或防抖  

工作量通常 **小于** Jump 绑定，但是对账的输入源。

### 3. Chrome 生命周期对账（中高）

`Map` 与检测结果每 tick 对齐：

| 事件 | 行为 |
| --- | --- |
| 新 panel | 创建侧栏，写初始 frame |
| frame 变 | 更新 origin（跟拖动/缩放） |
| panel 没了 | 拆侧栏、丢掉该 Path |
| 短暂检测失败 | 防抖，避免狂闪 |

难点：

- Cancel 后进程仍在、窗已无 → 已有「可见性」逻辑，多实例时每 key 都要稳  
- 主线程上 N 次 `setFrame` / `orderFront` 的节奏  
- 与 app 激活/失活策略叠加（见下）  

### 4. 显示/隐藏策略（中 · 产品规则但有实现量）

两种常见实现：

**A. panel 在则侧栏在**（跟生命周期）  
- 实现简单：对账只看 S  
- 可能：用户切到浏览器，侧栏仍浮在桌面  

**B. 仅当对应宿主/面板前台时显示该侧栏**  
- 每个实例一套 frontmost 判断  
- 更接近现在「让路」，但 N 倍分支  

难度不在 API，在 **规则写清 + 手测矩阵**。

### 5. UI 状态拆分（低～中）

| 状态 | 处理 |
| --- | --- |
| 列表数据 | 保持单例，灌给所有侧栏 **同一 snapshot** |
| Path 字符串 | 移入侧栏实例字段 |
| 列表点击 / ★ | 事件来自哪个 view → 改哪个实例的 Path / 对谁 Jump |
| 菜单「聚焦 Path」 | 需要解析「聚焦哪一个实例」（frontmost 映射或列表） |

共享列表 **不是** 难点；**事件路由到实例** 才是。

### 6. 几何与多屏（中 · 已有债务会放大）

单侧栏已有贴边/多屏不完美之处。  
N 个侧栏：重叠 dialog、左右同时贴、屏幕安全区，问题 **线性变多**，算法不必重写，但 **手测组合** 变多。

### 7. 平台 API 风险（低）

- 多 `NSPanel` floating：常规  
- AX 打指定 PID：已有能力  
- 不要求新的私有 API  
- Finder/zoxide 源与侧栏个数 **解耦**，可不动  

---

## 建议实现顺序（降低翻车概率）

1. Jump API **先** 参数化 target（单侧栏也走显式 target）——可单独合入、单测/手测  
2. 检测改为返回 **数组**（UI 仍只用 [0]）  
3. 引入 Map 对账，先允许 N 侧栏但 **hide 用简单策略 A**  
4. Path 每实例 + 列表事件路由  
5. 再打磨 hide 策略 B、Focus 菜单、防抖  

不要第一步就「复制 N 份控制器大泥球」。

---

## 测试成本（工程视角）

| 类型 | 单侧栏 | 多侧栏 |
| --- | --- | --- |
| 单元：检测集合、Map 对账 | 中 | 中高（状态机） |
| 单元：Jump 带 target | 可测 mock | 必须有 |
| HITL：双 Open 并排 Jump | 几乎不测 | **必测** |
| HITL：一个 Cancel 一个留 | 简单 | 必测 |
| 自动化 UI | 本来就难 | 更难 |

**结论：** 代码量中等，**验证成本** 明显高于「再加一个列表数据源」。

---

## 和「加一个 path 源」比难度

| 工作 | 相对难度 |
| --- | --- |
| 新 tab 数据源（如 zoxide） | 低：I/O + 列表 UI |
| 单侧栏 UX polish | 低～中 |
| **多侧栏** | **中：架构面 + 正确性 + HITL** |

多侧栏更重在 **并发目标与生命周期**，不是业务功能堆叠。

---

## 可给评审工程师的直接问题

1. 在「检测已能枚举多个 panel、Jump 已能打指定 PID」的前提下，你估 **Map 对账 + 多 panel 几何** 要几天？主要风险是否同意是 **串台 Jump**？  
2. panel key 你倾向 **PID 级** 还是 **PID+window**？系统 Open/Save 服务模型下哪种稳？  
3. 防抖/宽限：检测一帧丢目标时，你是否会要求侧栏延迟销毁？  
4. 若只能砍范围：是否接受 **先做 Jump 参数化 + 检测数组**，UI 仍单侧栏，作为多侧栏前置？  

---

## 一句话总结

**多侧栏在技术上可行、属中等难度重构；难点是目标绑定与生命周期对账，不是 UI 控件个数。列表全局共享反而简单。最大工程风险是静默 Jump 串台与双开场景的手测负担。**
